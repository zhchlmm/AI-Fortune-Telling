## Context

当前仓库未建立业务代码，仅具备 OpenSpec 流程骨架。目标是在不引入过多业务复杂度的前提下，先搭建“微信小程序 + 管理后台 API + 管理后台前端”三端可运行基础工程，并打通占卜会话主流程最小闭环。
约束包括：采用 ABP 模块化与 DDD 边界、前后端分离、中文文档规范、后续可演进到支付与运营能力。

## Goals / Non-Goals

**Goals:**
- 建立三端统一工程结构与可启动开发环境。
- 定义并实现占卜会话最小 API 契约（创建会话、查询会话、查询历史）。
- 管理后台具备占卜模板与内容资讯的最小管理入口。
- 小程序具备首页、占卜输入、结果展示、历史与个人中心基础页面。
- 建立环境变量模板与开发说明，支持本地联调。

**Non-Goals:**
- 不在本变更中实现完整 AI 推理质量优化与复杂算法引擎。
- 不在本变更中实现完整支付通道对接（仅保留订单接口与状态流转占位）。
- 不在本变更中实现复杂 BI 报表、活动运营系统与推荐系统。

## Decisions

- 决策1：采用 Monorepo 目录分层 `apps/*` 与 `services/*`。
  - 备选：分仓库管理。
  - 选择理由：当前处于 0→1 阶段，Monorepo 更利于统一契约、快速联调与跨端协同。

- 决策2：API 侧以 ABP 模块边界组织，先落地 ApplicationContracts + HttpApiHost 的最小闭环。
  - 备选：先用轻量 WebAPI 再迁移 ABP。
  - 选择理由：避免二次迁移成本，直接与既定技术栈一致。

- 决策3：占卜能力通过“模板配置 + 会话记录”模式解耦。
  - 备选：将模板硬编码在小程序端。
  - 选择理由：模板可由后台运营迭代，减少小程序发版频次。

- 决策4：首期占卜类型固定两类（塔罗、星座）作为混合能力起点。
  - 备选：一次性支持多种命理体系。
  - 选择理由：控制实现复杂度，先验证用户链路与留存。

## Risks / Trade-offs

- [风险] .NET 10 与 ABP 最新版本兼容性存在不确定性 → [缓解] 首批仅创建基础层，依赖版本在实现阶段锁定并验证。
- [风险] 小程序端 UI 组件与接口契约调整频繁 → [缓解] 先定义 DTO 与 mock 响应，页面按契约驱动开发。
- [风险] 支付相关合规与通道细节未确认 → [缓解] 仅做订单域占位，不触发真实扣款流程。
- [风险] 无历史代码参考导致命名分歧 → [缓解] 在本变更中固定目录与命名约定并文档化。

## Migration Plan

1. 在仓库创建三端目录骨架与初始化文件，不影响现有 OpenSpec 结构。
2. 增加后端最小 API 与健康检查接口，先本地单进程启动。
3. 接入前端与小程序的 API Base URL 配置，完成本地联调。
4. 若需回滚，仅删除新增目录与本变更文档，不影响历史数据（当前无生产数据）。
